# 测试场景与验收标准

## 使用说明

每个场景设计为一句自然语言指令，直接发送给 Claude Code 或 Codex。观察编排器是否正确：
1. 识别任务性质和规模
2. 选择合适的架构模式
3. 组建正确的团队
4. 按工作流执行
5. 产出符合预期的结果

---

## 场景一：新功能开发（流水线 + 生成-验证）

### 需求

```
组建团队，为一个 FastAPI 项目添加用户认证模块：
- JWT Token 认证
- 注册/登录/刷新 Token 三个接口
- 密码使用 bcrypt 加密
- 用 SQLAlchemy + SQLite 存储用户数据
```

### 预期行为

| 检查项 | 预期 |
|--------|------|
| 任务性质 | 新功能开发 |
| 规模判定 | 中等（4 个关注点：认证逻辑、数据模型、API 接口、安全） |
| 架构模式 | 流水线 + 生成-验证 |
| 团队组成 | architect + developer + reviewer（+ tester 可选） |
| Phase 1 | 向用户确认分析结果后再执行 |
| architect 产出 | `01_architect_design.md` — 含模块划分、接口定义、数据模型 |
| developer 产出 | 可运行的代码 + `02_developer_changelog.md` |
| reviewer 产出 | `03_reviewer_report.md` — 含安全性审查意见 |
| 执行顺序 | architect → developer → reviewer → developer（修改） |

需要对比 Click、Typer、Argparse、Fire 四个框架，

|--------|------|

| 架构模式 | 扇出/扇入 |

## 场景三：代码审查（扇出/扇入）

### 预期行为

| 检查项 | 预期 |
|--------|------|
| 任务性质 | 代码审查 |
| 规模判定 | 中等（3 个审查维度） |
| 架构模式 | 扇出/扇入 |
| 团队组成 | reviewer × 3（安全/可移植性/健壮性） |
| 并行执行 | 3 个 reviewer 独立并行审查 |
| 产出物 | 各自的 `03_reviewer_report.md` + 汇总报告 |
| 报告质量 | 按严重程度排序，每项有具体行号和修复建议 |

## 场景四：大规模重构（监督者）

### 需求

```
组建团队重构一个 Express.js 项目：
- 当前是单文件 app.js（2000 行），需要拆分为模块化结构
- 路由拆分到 routes/ 目录
- 中间件拆分到 middleware/ 目录
- 数据库操作拆分到 models/ 目录
- 工具函数拆分到 utils/ 目录
- 保持所有现有功能不变，确保拆分后可运行
```

### 预期行为

| 检查项 | 预期 |
|--------|------|
| 任务性质 | 重构 |
| 规模判定 | 大（6+ 个关注点：路由、中间件、数据库、工具、测试、整合） |
| 架构模式 | 监督者 |
| 团队组成 | project-lead + developer × 2-3 |
| project-lead 行为 | 分析依赖关系 → 分批分配 → 动态再分配 |
| developer 分工 | 按模块分配（如 developer-A 负责 routes + middleware，developer-B 负责 models + utils） |
| 产出物 | 重构后的模块化代码 + `02_developer_changelog.md` |
| 验证 | 拆分后代码结构正确，无循环依赖 |

# 验收评估标准

## 评分维度

对每个测试场景，按以下 6 个维度评分（每项 1-5 分）：

### 1. 任务分析准确性（Phase 1）

| 分数 | 标准 |
|------|------|
| 5 | 准确识别任务性质、规模、模式，团队方案最优 |
| 4 | 基本正确，团队方案合理但非最优 |
| 3 | 任务性质正确但规模或模式有偏差 |
| 2 | 多个维度判断有误 |
| 1 | 完全错误的分析 |

### 2. 团队组建合理性（Phase 2）

| 分数 | 标准 |
|------|------|
| 5 | 角色选择精准，规模合适，依赖关系清晰 |
| 4 | 角色合理，可能多了或少了一个非关键角色 |
| 3 | 缺少关键角色或冗余角色过多 |
| 2 | 团队组成与任务严重不匹配 |
| 1 | 未组建团队或组建完全错误 |

### 3. 执行流程规范性（Phase 3）

| 分数 | 标准 |
|------|------|
| 5 | 严格遵循架构模式，agent 间协调有序，产出物命名规范 |
| 4 | 基本遵循流程，有轻微偏离但不影响结果 |
| 3 | 流程有明显跳步或顺序错误 |
| 2 | 多数步骤未遵循预定流程 |
| 1 | 无流程可言，混乱执行 |

### 4. 产出物质量（Phase 4）

| 分数 | 标准 |
|------|------|
| 5 | 所有产出物完整、准确、格式规范，可直接使用 |
| 4 | 产出物基本完整，有小瑕疵但不影响使用 |
| 3 | 部分产出物缺失或质量不足 |
| 2 | 多数产出物不可用 |
| 1 | 无有效产出物 |

### 5. 协作效果

| 分数 | 标准 |
|------|------|
| 5 | agent 间信息传递准确、冲突处理得当、无重复工作 |
| 4 | 协作基本顺畅，偶有信息传递不完整 |
| 3 | 存在明显的协作断裂（信息丢失、重复工作） |
| 2 | agent 各自为政，缺乏有效协作 |
| 1 | 协作完全失败 |

### 6. 用户确认与透明度

| 分数 | 标准 |
|------|------|
| 5 | Phase 1 主动向用户确认，执行过程透明，最终报告清晰 |
| 4 | 有确认步骤但信息不够充分 |
| 3 | 跳过确认直接执行，但结果尚可 |
| 2 | 无确认且执行方向偏离用户意图 |
| 1 | 完全黑箱操作 |
